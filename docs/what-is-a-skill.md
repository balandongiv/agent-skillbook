# What Is a Skill?

This document explains the concept of a **skill** in this repository — what it is, how it is stored, and why the format is designed the way it is.

---

## The core idea

A **skill** is a document that teaches an AI agent how to handle a specific class of tasks. When the agent encounters a relevant request, it reads the skill's instructions and follows them.

Skills in this repository have two forms:
1. **Canonical form** — the human-editable source files
2. **Exported form** — auto-generated files formatted for a specific AI platform

---

## Canonical skill vs exported skill

### Canonical skill

The canonical skill is the **source of truth**. It lives in `skills/<slug>/` and consists of several files:

| File | Purpose |
|---|---|
| `skill.yaml` | Metadata, routing config, platform settings |
| `INSTRUCTIONS.md` | The actual instructions the agent will follow |
| `EXAMPLES.md` | Before/after examples showing the skill in action |
| `TESTS.md` | Test prompts to verify the skill is working |
| `CHANGELOG.md` | History of changes |
| `resources/notes.md` | Extra notes, references, research |

You edit canonical files. You never edit exported files directly.

### Exported skill

Exported files live in `skills/<slug>/exports/` and are generated by the renderer scripts in `tools/`. There is one export per AI platform:

```
exports/
├── openai/
│   ├── SKILL.md         ← Loaded by OpenAI Codex as a custom instruction
│   └── agents/
│       └── openai.yaml  ← OpenAI agent definition
├── claude/
│   └── SKILL.md         ← Loaded by Claude Code as a custom instruction
└── gemini/
    ├── GEM_INSTRUCTIONS.md  ← Text to paste into Gemini Gem UI
    └── SETUP.md             ← Step-by-step UI setup guide
```

Each platform expects a slightly different format, so the exported files are tailored to each.

---

## Why separate source from exports?

Several reasons:

1. **Each platform has a different format.** OpenAI, Claude, and Gemini each want frontmatter fields formatted differently, with different field names and values. Maintaining three versions manually would be error-prone.

2. **Single source of truth.** When you update `INSTRUCTIONS.md`, you regenerate all exports in one step. You cannot accidentally update one platform's export and forget another.

3. **Validation is easier.** The validator only needs to check canonical files. It does not need to parse platform-specific formats.

4. **Cleaner history.** Git diffs on canonical files are readable. If you edit raw exported files, the diffs become noisy with formatting changes.

## The skill.yaml format explained

Every field in `skill.yaml` has a specific purpose:

```yaml
slug: good-function-design
```
The unique identifier. Must match the folder name. Lowercase kebab-case only.

```yaml
title: Good Function Design
```
Human-readable name shown in lists and UIs.

```yaml
summary: Write small, readable, testable Python functions...
```
**This is the most important field.** The summary is used as the routing trigger for OpenAI and Claude — when a user's message matches this description, the agent will load this skill. See [description-writing-guide.md](description-writing-guide.md) for how to write good summaries.

```yaml
when_to_use:
  - when writing a new function
  - when refactoring a large function
```
A list of situations where this skill applies. Helps the agent decide when to activate the skill.

```yaml
when_not_to_use:
  - when the task is architecture-level design
  - when only formatting changes are needed
```
A list of situations where this skill should NOT be used. Prevents false positives.

```yaml
tags:
  - python
  - functions
  - readability
```
Keywords for filtering and discovery.

```yaml
invocation:
  auto: true
  explicit: true
```
- `auto: true` — the agent can trigger this skill automatically based on the description
- `explicit: true` — the user can invoke this skill by name

```yaml
platform_overrides:
  claude:
    disable_model_invocation: false
    user_invocable: true
    allowed_tools: []
  openai:
    include_openai_yaml: true
  gemini:
    manual_selection: true
```
Per-platform configuration. These values are passed through to the generated export files.

## The INSTRUCTIONS.md file

This is the heart of the skill. It contains the actual instructions the AI agent reads and follows. Write it in plain English, as if you are explaining a standard to a capable assistant.

Good instructions:
- Are specific and actionable
- Explain the *why* as well as the *what*
- Include rules, not vague advice
- Are organized with clear headings

---

## The EXAMPLES.md file

Each example should show:
1. A situation before the skill is applied (bad or naive approach)
2. The same situation after the skill is applied (correct approach)
3. A brief explanation of why the second version is better

Examples make the skill concrete and help the agent generalize to new cases.

---

## The TESTS.md file

Test prompts are natural language phrases a user might type into an AI agent that should trigger this skill. They are used to verify the skill is correctly configured and that the routing description is accurate.

Example test prompt: "Help me refactor this 80-line Python function."

If this prompt does NOT trigger the `good-function-design` skill, the description may need to be updated.
